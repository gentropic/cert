How to trust a GCU certificate

This page explains, in plain language, what a certificate issued by the Geoscientific Chaos Union actually claims, what mechanisms back each claim, and what it does not claim. The goal is that a curious non-specialist can read it in five minutes and know exactly what they are looking at.

What this certificate is

A GCU certificate is a signed statement from the Geoscientific Chaos Union that a named person completed a named workshop on a named date. Nothing more.

It is not a degree. It is not state-accredited. It is not a formal qualification recognized by any ministry of education. In Brazil it sits under the “curso livre” legal category (non-formal professional education); in the United States it is an unregulated “certificate of completion”; in the European Union it falls under non-formal learning outside the EQF unless a member state maps it explicitly. None of those contexts confer academic credit or professional licensing.

What the certificate is useful for: professional community recognition, continuing professional development (CPD) self-declaration under a member society’s rules, and structured evidence of workshop attendance that an employer or collaborator can verify independently. What it is not useful for: anything requiring formal academic credit.

How to verify a certificate

The fastest path is to click the QR code on the certificate (or open the validator URL at the bottom) and the validator page will run every check below in your browser and tell you the result. All verification is client-side: nothing is sent to GCU.

The authoritative artifact is the signed JSON credential, not the PDF. The PDF is a human-readable rendering of the same content, and also carries the JSON inside it as a PDF/A-3 associated file — your PDF reader’s attachment panel will show it. Any OBv3-compatible verifier (1EdTech reference validator, Digital Bazaar, Walt.id) can verify the JSON independently. GCU is not in the verification loop.

The trust stack

Five independent mechanisms back a GCU certificate. Each answers a different question. Any one of them failing is a red flag.

1

Issuer signature (Ed25519 / eddsa-rdfc-2022)

The credential is signed by GCU’s Ed25519 private key. Byte-level tampering breaks the signature. Forgery requires the private key, which is held as a GitHub Actions secret and mirrored in hardware-backed offline storage.

Your browser verifies the signature against the public key published at did:web:gentropic.org. If the DID document ever changes, the old key stays listed so past credentials keep verifying.

2

Self-endorsement

Alongside the credential, GCU emits a signed EndorsementCredential — a separate signed statement where the issuer endorses the authenticity of the credential and the relevance of the skills demonstrated. This is self-endorsement, not third-party endorsement, and is labeled as such. It sets up the schema so an external endorser (a university, a professional society) can sign their own endorsement later without any change to the data model.

3

Revocation (W3C BitstringStatusList)

Each credential carries an index into a public bit list. Bit 0 means “valid”; bit 1 means “revoked.” To revoke, GCU flips the bit, re-signs the list, and commits the change to this repository. The validator checks the list live and shows you the current state.

Honest limitation: revocation only works if the verifier can fetch the list. The list is mirrored to the Internet Archive monthly, and because it is a signed document, its authenticity doesn’t depend on where the bytes come from. This failure mode (revocation unavailable → valid-with-warning) is shared with every federated revocation system, including TLS and GPG.

4

Public issuance ledger

Every credential GCU has ever issued is recorded as one line in a hash-chained JSONL file in this repository. Each line commits to the previous one, so silently inserting or modifying a historical entry would break every subsequent line’s hash chain.

The validator checks that the credential’s hash appears in the ledger at the claimed index. A credential GCU refuses to acknowledge in the ledger is evidence of foul play, not evidence of validity.

5

Rekor transparency log (Sigstore)

Each credential’s hash is logged to Sigstore’s public Rekor transparency log at issuance time. Rekor is operated by the OpenSSF (Linux Foundation), audited by Trail of Bits, and is infrastructure GCU doesn’t control. The inclusion proof is committed alongside the credential as a .rekor.bundle file.

Even if GCU’s own infrastructure disappears, the Rekor record independently witnesses that the credential existed by the time it was logged. You can search by credential hash on search.sigstore.dev.

What this certificate does NOT guarantee

Equally important, because this is where well-intentioned readers get the wrong impression.

  • No accreditation. GCU is not a state-recognized educational institution.
  • No academic credit. The certificate confers no degree, diploma, or transferable credit.
  • No professional licensing. It does not qualify anyone for regulated professions (engineering, medicine, law, accountancy, etc.).
  • No guarantee of employability. It’s evidence of attendance; hiring decisions are the employer’s.
  • No substitute for formal education where that is legally required.

The certificate’s standing comes from GCU’s reputation and from the publicly auditable trust stack above. That is a modest basis and we want it to be read as such.

Who GCU is

The Geoscientific Chaos Union is a small collective exploring browser-native scientific computing, geoscience, and computational curiosity. Certificates are issued under the signing authority of Arthur Endlein Correia, the named primary signer for the current GCU workshops. The Ed25519 signing key is a GitHub Actions secret plus offline backup, never in a browser or on a laptop.

Workshops are self-serve: you open the workshop page, complete it at your own pace, and submit a completion hash shown on the claim form. A GitHub Actions workflow validates the hash against a salt only the workshop page knows, signs the credential, and commits it to this repository. There is no human review per credential. The trust story for that flow is written out in policies/issuance.md.

Fork your own

The entire system is open source, including this page and the validator. A future v2 release will extract the engine and design into a reusable template so forking an instance becomes a configuration exercise; as of this writing you would need to read the spec and implementation plan and follow the phases.
